# Release verifiers

`verifiers` versions are git-driven via `hatch-vcs`: there is no version string to edit. A pushed `vX.Y.Z` tag is the
only thing that cuts a stable release — the `Publish verifiers` workflow (`.github/workflows/publish-verifiers.yml`)
then builds the tag, publishes to PyPI via OIDC, and creates a GitHub Release with auto-generated notes. Every push to
`main` already publishes a `0.1.X.dev<N>` pre-release automatically; do not tag dev versions.

Drive the release with the `gh` CLI.

## Steps

```
- [ ] 1. Choose the version
- [ ] 2. Confirm main is green
- [ ] 3. Tag and push
- [ ] 4. Watch the workflow
- [ ] 5. Verify + (optionally) curate notes
```

### 1. Choose the version

Look at the latest stable tag and pick the next `vX.Y.Z` per the release policy (`assets/release/release_workflow.md`):

```bash
git fetch --tags
git tag --list 'v[0-9]*' | grep -vE 'dev|rc' | sort -V | tail -1
```

- patch (`Z+1`): bug-fix / model-only release
- minor (`Y+1`): regular feature release
- major (`X+1`): sweeping API change

### 2. Confirm main is green

Release from a green commit on `main`:

```bash
gh run list --branch main --limit 5
```

### 3. Tag and push

Tag the intended commit and push. The push is the trigger, so do this exactly once per version.

```bash
git tag -a vX.Y.Z -m "Release vX.Y.Z"
git push origin vX.Y.Z
```

Do NOT run `gh release create` — the workflow creates the GitHub Release. Creating it manually races the workflow.

### 4. Watch the workflow

```bash
gh run watch "$(gh run list --workflow=publish-verifiers.yml --event=push --limit=1 --json databaseId --jq '.[0].databaseId')"
```

Confirm the `build-tag`, `publish-tag`, and `github-release-tag` jobs succeed (publish runs in the `pypi-prod`
environment via OIDC).

### 5. Verify and optionally curate notes

```bash
gh release view vX.Y.Z --web      # confirm artifacts + auto-generated notes
```

Release notes are auto-generated by GitHub from merged PRs (a "What's Changed" list) — they are not kept in the repo.
To add a short curated "Highlights" section before announcing, edit the existing release in place:

```bash
gh release edit vX.Y.Z --notes-file -   # paste curated notes on stdin
```

## Re-running a failed release

The tag already exists, so re-trigger via dispatch instead of re-tagging:

```bash
gh workflow run publish-verifiers.yml -f tag=vX.Y.Z
```

If PyPI rejected a partial upload, remove the partially uploaded files for that version on PyPI before retrying.
